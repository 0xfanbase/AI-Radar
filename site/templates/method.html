Method

Every story on this site moves through the same fixed pipeline before it ever reaches this page, and the pipeline itself is entirely code -- watching, drafting, and fact-checking are each a separate, deliberately narrow step, and no single step can skip the ones before it.

1. Finding what's worth covering

A daily, fully automated watcher pulls from a fixed set of sources -- frontier labs' own blogs and release pages, arXiv, and Hacker News -- clusters near-duplicate reports of the same story together, and ranks the result by how many independent sources are covering it and how much attention it's getting. Only the top handful of clusters each day ever reach the next step; nothing is invented, and an empty result on a quiet day is treated as a normal outcome, never a bug to route around.

2. Drafting and fact-checking

An AI analyst drafts each story from the sources the watcher found, always in its own words, with short attributed quotes only. A story only publishes as CONFIRMED if it's backed by the lab's own announcement or by two independent, reputable outlets reporting it separately; short of that bar it publishes as REPORTED, with its sourcing shown plainly, or it doesn't publish at all. A second AI pass -- the verifier -- then re-checks the draft from scratch, in a fresh context, re-fetching every citation and re-reading it independently. The verifier can strip an unsupported sentence, drop a hollowed-out story entirely, or demote a story's status, but it can never upgrade one -- it's a one-way ratchet toward stricter, never looser.

Each tracked lab's profile page goes through a related but smaller process. A plain-code step picks at most one company to touch each day -- either a lab named in that day's own stories, or, if none is, whichever tracked profile has gone longest without being re-verified, once that exceeds a fixed 45-day floor -- so no profile sits unchecked indefinitely even in a quiet news week. A separate AI role then fetches that one company's own pages plus independent outlet coverage and applies the exact same CONFIRMED/REPORTED sourcing bar as the card pipeline above, updating only the fields new sourcing actually supports. Unlike a story card, a profile page does not get a second, fresh-context adversarial pass -- it is checked once, by the same role that drafts it, not re-verified by an independent second reader. That is a real, narrower guarantee than a card gets, stated plainly rather than implied.

3. Placing companies on the map

The homepage map plots one marker per tracked lab, positioned at the headquarters coordinates recorded on that lab's own profile -- latitude and longitude sourced the same way any other fact on this site is, not estimated or auto-geocoded from a company name. A handful of real-world clusters (multiple labs headquartered within a few kilometres of each other) are nudged apart by a small, fixed, hand-picked pixel offset purely so their markers don't visually overlap at world-map scale; the offset never changes what coordinates are actually on record. Clicking a company's name always goes straight to its profile page, with no script required. A second, separate control on each marker -- a small glyph, not the name itself -- expands a popover of that company's latest Frontier Board entries and most recent wire stories; that popover interaction is the site's only other JavaScript beyond the animated background, and it only ever runs on this one page. If it fails to load for any reason, every marker's name link and the plain company list still work exactly as before.

4. A gate neither AI step can get past

Before anything is published, a plain-code check confirms every changed file is schema-valid and that the day's change only ever touched the site's editorial content and pipeline-state folders -- never the workflows, the pipeline's own code, or the rules on this page. If that check fails, nothing commits. This is also the concrete reason a hostile instruction hidden in a fetched web page can't do anything more than get treated as an ordinary quote to fact-check: at the absolute worst, it could influence the wording of one story, never the system that produces it.

5. Keeping outbound links honest

Every outbound link this site cites -- in a story or a company profile -- is checked against a small, fixed, human-curated list of trusted destinations before it can ever be published: the domain has to match the list outright, plain http:// links and bare IP addresses are rejected, and the link's actual redirect chain is followed and checked too, so a citation that looks trusted but quietly forwards somewhere else can't slip through. That same allowlist file is frozen -- the automated pipeline that writes stories and profiles is never allowed to edit it, so it can never expand its own linking budget on the same run it uses a new domain. A separate weekly pass re-resolves every citation already published, catching the case where a domain was trustworthy when first cited but has since been abandoned, sold, or repointed elsewhere, and flags any company profile that has gone more than 45 days without being re-checked against its own sources for the daily process above to pick up.

Stated plainly, because it matters what this mechanism does and doesn't do: it checks where a link points -- the domain and the real destination of its redirect chain -- against a trusted list. It does not scan the destination page's own content for malware, does not evaluate whether a trusted domain's specific page is itself safe or accurate, and offers no protection at all against a citation to a site that was compromised without changing domains or redirecting anywhere. It is a link-provenance check, not a content or malware scanner, and this page will not describe it as more than that.

Once running, this weekly, entirely pure-code pass checks every published citation -- in both stories and company profiles -- for link rot and for the redirect-hijack check described above, checks whether every AI/ML term actually used in a story is defined in the Lexicon (and flags any Lexicon term nothing has used in a while), tracks the verifier's own pass rate over time, compares published stories against that week's most-discussed Hacker News AI stories to catch genuine misses, looks for duplicate coverage of the same topic, and flags any company profile past the 45-day re-verification floor. Findings are logged as findings, not failures -- a story correctly declined for weak sourcing is not the same thing as a story the pipeline missed. A stale-profile or hijacked-citation finding is also queued for the next day's profile-refresh pass to act on, the same way any other correction candidate is.
